Problem

課程內容越完整，學生越需要一個可信的入口。

教材分散

PDF、Markdown、公告、FAQ、Q&A 與講義會散落在不同格式。單純全文搜尋很難回答跨章節問題。

回答必須有根據

學生不只需要自然語言答案，也需要知道答案來自哪個檔案、哪個 heading、哪些片段。

營運需要可控

上傳、索引、權限、模型成本、錯誤、備份與部署，都要能被追蹤，而不是藏在臨時腳本裡。

Product Promise

不是聊天機器人外殼，而是可審計的課程知識產品。

專案把「回答」拆成三個可檢查的合約：內容如何進來、證據如何被選出、引用如何被驗證。

1

Canonical sources

所有輸入最後落到 canonical Markdown，引用回到穩定的 `filename#heading`。

2

Grounded retrieval

Postgres lexical search 與 pgvector semantic search 合併排序，保留診斷資料。

3

Citation validation

答案只能引用 selected sources，否則降級成穩定的 cannot-confirm 回覆。

4

Operator control

Admin key、audit events、worker heartbeat、provider budget 與 smoke checks 都納入產品面。

End-to-End Flow

從教材檔案到 grounded answer，每一步都留下可營運的狀態。

01UploadPDF / MD / TXT / HTML through `/imports`
02ValidateReject empty, binary-looking, bad PDF, HTML mismatch
03CanonicalizeRaw artifact plus canonical Markdown in `docs/`
04IndexDocuments, sections, chunks, embeddings, TSV
05RetrieveVisibility-filtered lexical + vector merge
06AnswerCitation validation before persistence

關鍵設計：`POST /imports` 保持輕量，真正的轉檔、索引、eval 交給 DB-backed background worker。

Data Model

資料表設計讓來源、回答、診斷與營運狀態可以分開追蹤。

documentsCanonical path、source type、title、content hash、visibility、lifecycle status 與 metadata。
sections / chunksMarkdown headings 變 sections；sections 有 TSV，chunks 有 `vector(768)` HNSW index。
conversations / messages保存 learner 與 assistant messages，assistant message 保留 cited sources JSON。
retrieval_events保存 selected sources、score diagnostics、answer quality、provider metadata 與 decision。
background_jobs / audit / feedbackDB-backed orchestration、security audit、learner/TA feedback 與 eval regression loop。
Ingestion

匯入層的目標不是只把檔案存起來，而是保證 runtime source 一致。

原始上傳留在 `raw/`，可引用的知識留在 `docs/` canonical Markdown。同內容用 hash 去重；同檔名不同內容用 hash suffix 保留新 artifact。

.pdf

檢查 PDF signature，抽文字轉 Markdown。

.md

保留 Markdown，補 metadata 與 canonical path。

.txt

UTF-8 解碼、段落化、簡單 heading detection。

.html

要求 recognizable markup，轉成可讀 Markdown。

Reject

空檔、content-type mismatch、binary-looking text。

Metadata

原檔名、檔案類型、warnings、Markdown bytes。

Async

`POST /imports` 建 queued job，worker 執行轉換。

Next job

成功後 enqueue `index.rebuild`。

Indexing

Index 不是檔案快取，而是回答品質與引用可靠性的結構化基礎。

1Scan `docs/`掃描 canonical Markdown，frontmatter 可帶 visibility labels。
2Parse sections用 heading 建 source IDs，保留 heading slug、level、body_md。
3Chunk and embed只對變更 chunks 建 embeddings；目前 schema dimension 是 768。
4Export status寫入 Postgres index，也產生 `.kb/index.json` 供 inspection/export。
$ make migrate
$ make index
$ make worker-once
ready: database, pgvector, migrations, index, platform_auth, storage
Retrieval Contract

Hybrid retrieval 會產生答案，也會產生可解釋的診斷資料。

Search path 先做 source access filtering，再跑 lexical / vector / hybrid。合併後套 query relevance boost、source priority boost、score threshold，最後選出 accepted candidates。

Lexical

Postgres TSV 搜尋 headings 與 Markdown body，適合精確術語、檔名、章節。

Vector

pgvector cosine search，適合語意相近、不同說法、跨教材提問。

Hybrid merge

依 section 合併候選，保留 strategy scores 與 debug scores。

Evidence gate

低於 threshold 或沒有 accepted source，就回 cannot-confirm。

Answer Quality

回答層的核心規則：不能引用未選中的來源。

Selected sources

只有 retrieval accepted candidates 會進 prompt。這是模型可用證據的邊界。

Cited sources

LLM 產出的引用會被解析，必須全部屬於 selected source IDs。

Invalid citations

若漏引、亂引、cannot-confirm 卻附引用，就 retry strict；仍失敗則降級。

answer_quality = {
  "answer_valid": true,
  "selected_source_ids": [...],
  "cited_source_ids": [...],
  "cannot_confirm_reason": null
}

Access Control

平台登入是學員入口，Admin key 是營運入口，兩者刻意分離。

MVP 不做註冊或完整 RBAC，而是單一 configured platform login。內容可見性透過 source labels 套用在 search、chat、sources、mindmap。

Platform learnerSession cookie with `KB_AUTH_SECRET_KEY`; 可看 `public`、`role:*`、`user:*`、`cohort:*` 與額外 labels。
Admin operations`X-KB-Admin-Key` 保護 uploads、indexing、jobs、audit、documents、provider ops。
Document lifecycle`active` 才能被 sources/search/chat/mindmap 看見；`delete` 只刪 DB index，不刪原檔。
Secret hygieneAudit metadata 不存 raw admin keys 或 platform passwords；admin actor ID 用 fingerprint。

Operations

這個 MVP 的 observability 是 app-native，先滿足單機部署與快速 triage。

/health + /ready

Ready 檢查 DB、pgvector、migration、index、auth、storage。

/metrics

In-process JSON counters、recent request samples、provider usage 與 error metadata。

Request IDs

每個 response 都帶 `X-Request-ID`，structured logs 可串 UI 回報與 server logs。

Rate limits

Login、chat、streaming chat、uploads、admin writes 都有 429 與 `Retry-After`。

Audit events

Login、logout、admin allow/deny、rate-limit block、upload concurrency block 入 DB。

Provider budget

Tokens、calls、error rate 可設 warning/exceeded；可選擇 exceeded 時阻擋 provider call。

Quality Loop

回答品質不是主觀感覺，而是 retrieval 與 citation 分開度量。

Eval metrics 明確區分找不到證據、找到但沒引用、引用不精準等失敗型態，方便後續調整檢索策略與教材結構。

Retrieval

`top1_hit`, `retrieval_recall`, selected/rejected source IDs, top score。

Citation

`citation_recall`, `citation_precision`, citation error count。

Answer validity

`answer_valid`, cannot-confirm reason, invalid provider citations。

Feedback

Learner/TA ratings and notes become regression inputs.

Trade-offs

這些取捨讓 MVP 可以上線，同時留下清楚的演進路徑。

Postgres + pgvector over FAISS

同一個 operational boundary 管 metadata、permissions、chunks、vectors、feedback、retrieval events。

Single platform login

先滿足課程學員入口，避免 premature user management；admin key 保持獨立。

App-native observability

先不引 Prometheus/OpenTelemetry，等流量或多 replica 真的需要時再加。

No file deletion on document delete

Document delete 是 DB index lifecycle 操作，raw/canonical artifacts 保留。

Provider budget guardrails

成本與錯誤率先在 app 中可見，必要時可 blocking before provider call。

Canonical Markdown first

讓 citations、source preview、mindmap、evaluator 使用同一份穩定文字來源。

Takeaway

這個系統的價值不只是回答比較快，而是讓每個答案都能回到課程材料、引用合約與營運事實。

For learners

更快找到可信答案。

For admins

更安全地匯入、索引、修復教材。

For operators

更清楚地部署、觀測、控制成本。
